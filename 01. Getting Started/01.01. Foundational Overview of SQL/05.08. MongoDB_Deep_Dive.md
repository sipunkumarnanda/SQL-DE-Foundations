# 📖 What is MongoDB?

## Definition

MongoDB is a **NoSQL Document Database** that stores data as **documents** instead of rows and columns.

Unlike relational databases,

MongoDB stores information in

JSON-like documents.

Internally,

MongoDB stores them in a binary format called

**BSON (Binary JSON).**

---

# 🏠 Real-Life Analogy

Imagine a filing cabinet.

Each customer has one file.

Every file contains everything about that customer.

```text
📁 Customer File

Name

Phone

Email

Address

Orders

Skills

Preferences
```

Instead of splitting the information into many tables,

MongoDB often keeps related information together in one document.

# 📂 What is a Collection?

## Definition

A **Collection** is a group of related documents.

It is similar to a table in SQL,

but more flexible.

---

# Example

Students Collection

```json
{
 "_id":1,
 "name":"Rahul"
}

{
 "_id":2,
 "name":"Priya"
}

{
 "_id":3,
 "name":"Ankit"
}
```

Each object is one document.

Together,

they form a collection.

---

# 📄 What is a Document?

A document is the smallest unit of data stored in MongoDB.

Think of it as one complete object.

Example

```json
{
 "_id":1,
 "name":"Rahul",
 "email":"rahul@gmail.com",
 "age":22,
 "city":"Mumbai"
}
```

Everything about Rahul is stored together.

---

# 🧠 What is a Field?

Fields are simply

key-value pairs.

Example

```json
{
 "name":"Rahul"
}
```

Here

```text
name
```

is the field.

```text
Rahul
```

is the value.

---

# 🤔 Why Doesn't MongoDB Need a Fixed Schema?

Suppose today

you only store

```json
{
 "name":"Rahul"
}
```

Tomorrow,

you want to add

GitHub.

Simply do

```json
{
 "name":"Rahul",
 "github":"rahul-dev"
}
```

Another user

```json
{
 "name":"Priya",
 "portfolio":"priya.dev"
}
```

Both documents are valid.

There is no requirement that every document has identical fields.

This flexibility is useful for rapidly evolving applications.

# 🆔 What is `_id`?

Every document in MongoDB has

```text
_id
```

This field uniquely identifies the document.

It is similar in purpose to a Primary Key in SQL.

Example

```json
{
 "_id":1,
 "name":"Rahul"
}
```

Every document must have a unique `_id`.

If you don't provide one,

MongoDB automatically generates it.

---

# 🤔 What is ObjectId?

Most MongoDB documents use

ObjectId

as the default `_id`.

Example

```text
684bcff1d8d9b65d5d93d6e2
```

It looks random.

But it isn't.

---

# 🧩 ObjectId Structure

An ObjectId is 12 bytes long and includes:

* Timestamp (creation time)
* Random value
* Counter for uniqueness

This design makes ObjectIds unique across many machines without requiring a central ID generator.

# 📦 Nested Documents

MongoDB allows documents inside documents.

Example

```json
{
 "name":"Rahul",
 "address":{
   "city":"Mumbai",
   "state":"Maharashtra"
 }
}
```

The address is stored within the same document.

---

# 📋 Arrays

MongoDB can store arrays directly.

Example

```json
{
 "name":"Rahul",
 "skills":[
   "Java",
   "React",
   "MongoDB"
 ]
}
```

In SQL,

this would typically require a separate table to model properly.

MongoDB can store the array directly when it fits the application's needs.

---

# 🛒 Example

Amazon Product

```json
{
 "name":"Laptop",
 "price":70000,
 "colors":[
   "Black",
   "Silver"
 ]
}
```

Arrays make it easy to represent multiple values in one document.

---

# 🔗 Embedded Documents

Instead of creating many tables,

MongoDB often embeds related data.

Example

```json
{
 "customer":"Rahul",
 "orders":[
   {
     "product":"Laptop",
     "price":70000
   },
   {
     "product":"Mouse",
     "price":500
   }
 ]
}
```

Customer and orders are stored together.

This can make reading related information very fast.

---

# ⚠️ Should Everything Be Embedded?

No.

Embedding is useful when:

✅ Related data is usually read together.

Referencing separate documents is often better when:

* The related data grows very large.
* Multiple documents need to share the same information.
* The embedded data changes frequently.

Choosing between embedding and referencing is an important design decision.

# 💾 How Does MongoDB Store Data Internally?

Many beginners think

MongoDB stores JSON files.

❌ Not exactly.

Internally,

MongoDB stores documents as

**BSON (Binary JSON).**

---

# 🤔 Why BSON Instead of JSON?

Binary formats are more efficient for computers to process.

BSON supports additional data types that standard JSON does not, such as:

* Dates
* Binary Data
* Decimal128
* ObjectId

This makes it more suitable for database storage.

---

# ⚙️ Internal Flow

Suppose we insert

```json
{
 "name":"Rahul"
}
```

The flow is approximately:

```text
Application

↓

JSON

↓

MongoDB Driver

↓

BSON

↓

Storage Engine

↓

Disk
```

Applications usually work with JSON-like objects,

while MongoDB stores BSON internally.

---

# 📚 Storage Engine

Modern MongoDB uses the

**WiredTiger**

storage engine by default.

Its responsibilities include:

* Reading documents
* Writing documents
* Compression
* Caching
* Recovery
* Transactions

The storage engine is similar in purpose to MySQL's InnoDB storage engine.

---

# 🌳 Indexes in MongoDB

Without an index,

finding one user among

10 million documents

may require scanning many documents.

Indexes allow MongoDB to locate documents much faster.

MongoDB primarily uses **B-tree-based indexes** (more precisely, B+ tree variants depending on the implementation), similar in concept to relational databases.

# 🎤 Interview Questions

## ❓ What is MongoDB?

### ✅ Answer

MongoDB is a NoSQL document database that stores data as BSON documents inside collections. It supports flexible schemas, nested documents, arrays, and horizontal scaling, making it well suited for applications with evolving data models.

---

## ❓ Why is MongoDB called a Document Database?

### ✅ Answer

MongoDB stores information as self-contained documents rather than rows in tables. Each document contains related fields together, often allowing applications to retrieve all required information with fewer operations.

---

## ❓ What is ObjectId?

### ✅ Answer

ObjectId is MongoDB's default unique identifier for documents. It is a 12-byte value containing components such as a timestamp, random data, and a counter, allowing unique IDs to be generated efficiently across distributed systems.

---

# 🎯 Key Takeaways

✅ MongoDB is a NoSQL document database.

✅ Collections correspond roughly to tables, and documents correspond roughly to rows.

✅ Documents are stored internally as BSON.

✅ MongoDB supports flexible schemas.

✅ Documents can contain nested objects and arrays.

✅ Every document has a unique `_id`, usually an ObjectId.

✅ WiredTiger is the default storage engine.

✅ MongoDB uses indexes to accelerate queries.

✅ MERN applications benefit from MongoDB's document model because it maps naturally to JavaScript objects.
