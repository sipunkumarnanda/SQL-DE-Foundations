

# 📚 Why is Horizontal Scaling Difficult in SQL Databases? (Complete Interview Notes)

---

# 🎯 Introduction

One of the most common interview questions is:

> **"Why are SQL databases difficult to scale horizontally?"**

Many people answer:

> ❌ "Because SQL databases don't support horizontal scaling."

This answer is **incorrect**.

The correct answer is:

> ✅ **SQL databases *do* support horizontal scaling, but it is much more difficult to implement than in most NoSQL databases because SQL databases guarantee ACID transactions, relationships, joins, and strong consistency.**

The challenge is **not adding more servers**.

The challenge is:

> **Keeping all servers synchronized while maintaining SQL guarantees.**

---

# 🚀 First Understand What Scaling Means

Scaling means increasing the capacity of your database to handle more users, more data, and more requests.

There are **two types of scaling.**

---

# 🖥️ 1. Vertical Scaling (Scale Up)

Vertical scaling means:

> **Make one server more powerful.**

Instead of buying more servers, you upgrade the existing one.

### Example

Before

```text
Database Server

CPU : 4 Cores
RAM : 16 GB
Storage : 500 GB SSD
```

⬇️ Upgrade Hardware

After

```text
Database Server

CPU : 32 Cores
RAM : 256 GB
Storage : 8 TB NVMe SSD
```

Notice:

The database is still running on **one machine**.

Only its hardware has improved.

---

## ✅ Advantages

✔ Very easy

✔ No application changes

✔ Existing SQL queries continue working

✔ No distributed communication

---

## ❌ Disadvantages

Eventually hardware reaches its limit.

Imagine:

```
Maximum CPU
Maximum RAM
Maximum SSD
```

You cannot keep upgrading forever.

Also,

If that single server crashes,

your entire database becomes unavailable.

---

# 🌐 2. Horizontal Scaling (Scale Out)

Instead of buying a larger server,

buy multiple servers.

Example

```
        Database Cluster

      🖥️ Server 1

      🖥️ Server 2

      🖥️ Server 3

      🖥️ Server 4
```

Now,

the database is distributed across many machines.

Each machine stores only part of the data.

---

## ✅ Advantages

✔ Almost unlimited scalability

✔ High availability

✔ Better fault tolerance

✔ Easy to add new servers

---

## ❌ Disadvantages

Very difficult to maintain correctness.

This is where SQL databases become complicated.

---

# 🤔 Why is Horizontal Scaling Difficult?

Let's understand using real examples.

---

# 📌 Problem 1 — JOIN Operations Become Difficult

Imagine we have two tables.

## Users Table

| id | name  |
| -- | ----- |
| 1  | Alice |
| 2  | Bob   |

---

## Orders Table

| id | user_id | amount |
| -- | ------- | ------ |
| 1  | 1       | 500    |
| 2  | 2       | 300    |

---

Suppose we execute

```sql
SELECT Users.name, Orders.amount
FROM Users
JOIN Orders
ON Users.id = Orders.user_id;
```

---

## 🟢 Case 1 : Single Server

Everything is stored in one computer.

```
CPU

↓

Users Table

↓

Orders Table

↓

JOIN

↓

Result
```

Everything happens inside RAM.

No network communication.

Very fast.

---

## 🔴 Case 2 : Distributed Database

Now imagine

```
🖥️ Server 1

Users Table
```

```
🖥️ Server 2

Orders Table
```

Now the same query becomes

```
Client

↓

Server 1

↓

Needs Orders

↓

🌐 Network Request

↓

Server 2

↓

Find Orders

↓

🌐 Send Data Back

↓

Server 1

↓

Perform JOIN

↓

Return Result
```

Notice what happened.

Earlier

```
CPU ↔ RAM
```

Now

```
CPU

↓

Network

↓

Another Computer

↓

Network

↓

CPU
```

Network communication is much slower than RAM.

So,

JOIN becomes expensive.

---

## 💡 Interview Point

SQL databases heavily depend on JOIN operations.

When related tables are stored on different servers,

JOIN requires network communication,

making queries slower and more complex.

---

# 📌 Problem 2 — ACID Transactions Become Very Complex

Imagine Alice transfers ₹100 to Bob.

Current Balance

```
Alice = ₹1000

Bob = ₹500
```

Transaction

```
1. Deduct ₹100 from Alice

2. Add ₹100 to Bob
```

Either both happen

OR

Neither happens.

This is called **Atomicity**.

---

## 🟢 Single Server

Everything happens inside one database.

```
BEGIN

↓

Deduct Alice

↓

Add Bob

↓

COMMIT
```

If power fails

```
ROLLBACK
```

Simple.

---

## 🔴 Distributed Database

Now imagine

```
🖥️ Server 1

Alice
```

```
🖥️ Server 2

Bob
```

Transaction

```
Server 1

↓

Deduct ₹100

↓

🌐 Network

↓

Server 2

↓

Add ₹100
```

Now imagine

Server 1 succeeds

but

Server 2 crashes.

Final result

```
Alice = ₹900

Bob = ₹500
```

₹100 disappeared.

The database is corrupted.

---

## 🤯 How SQL Solves This?

SQL databases use distributed transaction protocols.

For example,

### Two-Phase Commit (2PC)

### Phase 1 — Prepare

Coordinator asks every server

```
"Are you ready?"
```

Server 1

```
✅ Yes
```

Server 2

```
✅ Yes
```

---

### Phase 2 — Commit

If everyone agrees

```
COMMIT
```

Otherwise

```
ROLLBACK
```

This guarantees correctness,

but introduces:

❌ More network communication

❌ Waiting time

❌ Higher latency

---

## 💡 Interview Point

Distributed ACID transactions require coordination between multiple servers,

making horizontal scaling much harder.

---

# 📌 Problem 3 — Foreign Keys Become Difficult

Suppose

```
Users

id
```

Orders

```
user_id
```

Foreign Key

```
Orders.user_id

↓

Users.id
```

Whenever an order is inserted,

SQL checks

> "Does this user actually exist?"

---

## 🟢 Single Server

```
Check Users Table

↓

Done
```

---

## 🔴 Distributed Database

```
Orders

↓

Server 2
```

```
Users

↓

Server 1
```

Every insert becomes

```
Server 2

↓

🌐 Ask Server 1

↓

Does User Exist?

↓

Response

↓

Insert Row
```

Every request now requires network communication.

Slower.

---

# 📌 Problem 4 — Strong Consistency

Suppose

```
Balance = ₹1000
```

Two people update it simultaneously.

Person A

Deposits ₹500

Person B

Withdraws ₹200

---

Without synchronization

Server 1

```
1000

↓

1500
```

Server 2

```
1000

↓

800
```

Which one is correct?

Neither.

One update may overwrite the other.

This is called a **race condition**.

---

## SQL Solution

SQL databases use

✔ Locks

✔ MVCC (Multi-Version Concurrency Control)

✔ Timestamps

✔ Distributed consensus protocols

These techniques maintain correctness,

but increase complexity and overhead.

---

# 📌 Problem 5 — Sharding

Horizontal scaling usually uses

## Sharding

Meaning

Split data across servers.

Example

```
Server 1

Users A-H
```

```
Server 2

Users I-P
```

```
Server 3

Users Q-Z
```

---

Searching Alice

```
Find Alice

↓

Server 1

↓

Done
```

Very fast.

---

But now imagine

```sql
SELECT *
FROM Users;
```

Database must

```
Ask Server 1

↓

Ask Server 2

↓

Ask Server 3

↓

Merge Results

↓

Return Result
```

One query becomes many distributed queries.

---

# 📌 Problem 6 — Distributed Locks

Suppose

Two users want to buy

The last iPhone.

Inventory

```
iPhone = 1
```

Server 1

```
Reads

1 Available
```

Server 2

```
Reads

1 Available
```

Both sell it.

Now

Inventory becomes

```
-1
```

Impossible.

To prevent this,

distributed locks are required.

Distributed locks are much slower than local locks.

---

# 📌 Problem 7 — Replication Delay

Many SQL databases use replicas.

```
Primary Database

↓

Replica 1

↓

Replica 2
```

Suppose

You update your password.

Immediately afterward,

another request reaches Replica 2.

Replica 2 hasn't received the update yet.

It still returns old data.

This is called

**Replication Lag**.

---

# 📌 Problem 8 — Network Failures

Inside one computer,

RAM almost never "disconnects."

But servers communicate over a network.

Problems include:

📡 Slow network

🔌 Cable failures

🌩️ Packet loss

💥 Server crashes

SQL databases must detect and recover from all of these conditions while preserving correctness.

---

# 🤔 Why NoSQL Finds Horizontal Scaling Easier

Many NoSQL databases store related information together.

Example

```json
{
  "name": "Alice",
  "email": "alice@example.com",
  "orders": [
    {
      "id": 1,
      "amount": 500
    },
    {
      "id": 2,
      "amount": 700
    }
  ]
}
```

Everything about Alice is inside one document.

Reading Alice

↓

Reading Orders

↓

Done.

No JOIN required.

When this document is moved to another server,

everything moves together.

This makes sharding easier.

---

# ⚖️ Trade-Offs

## SQL

✅ Strong consistency

✅ ACID transactions

✅ JOINs

✅ Foreign keys

✅ Data integrity

❌ More difficult horizontal scaling

---

## NoSQL

✅ Easy horizontal scaling

✅ Flexible schema

✅ High write throughput

✅ Massive scalability

❌ Data duplication is common

❌ Complex JOINs are limited or avoided

❌ Some systems relax consistency for availability

---

# 🎤 Interview Answer (Best Version)

> SQL databases are not difficult to scale because they lack horizontal scaling support. They are difficult to scale because they guarantee ACID transactions, strong consistency, JOINs, and referential integrity. When data is distributed across multiple servers, these guarantees require constant coordination and communication over the network. Operations such as distributed JOINs, transactions, foreign key checks, locking, and synchronization become much more complex and expensive. NoSQL databases often simplify horizontal scaling by storing related data together and avoiding many cross-server operations, allowing them to scale out more easily for certain workloads.

# 📝 Final Revision Points

## ✅ Vertical Scaling

* One server
* Upgrade CPU, RAM, SSD
* Simple
* Hardware limit

---

## ✅ Horizontal Scaling

* Multiple servers
* Better scalability
* Better availability
* More complex

---

## 🚨 Why SQL Horizontal Scaling is Hard

🔹 JOINs require cross-server communication.

🔹 ACID transactions require distributed coordination.

🔹 Foreign keys need validation across machines.

🔹 Strong consistency requires synchronization.

🔹 Distributed locks add overhead.

🔹 Sharding complicates query execution.

🔹 Replication introduces synchronization challenges.

🔹 Network failures and latency must be handled safely.

---

# ⭐ One-Line Summary

> **SQL databases can scale horizontally, but preserving ACID properties, joins, consistency, and referential integrity across multiple servers requires distributed coordination, making horizontal scaling significantly more complex than in many NoSQL databases.**
