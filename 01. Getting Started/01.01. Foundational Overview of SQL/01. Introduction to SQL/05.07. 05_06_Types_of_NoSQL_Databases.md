# 🌍 Introduction

In the previous chapter, we learned that

**NoSQL** was created to solve new problems like:

✅ Flexible schemas

✅ Massive scalability

✅ Distributed systems

✅ High-speed workloads

But here's something many beginners don't know.

👉 **There isn't just one kind of NoSQL database.**

Instead,

NoSQL is a **family of database models**.

Each model stores data differently.

Each is optimized for different kinds of applications.

---

# 🏗️ Four Major Types of NoSQL Databases

There are four major categories.

```text
NoSQL Databases

│

├── 📄 Document Database

├── 🔑 Key-Value Database

├── 📊 Wide-Column Database

└── 🕸️ Graph Database
```

Every popular NoSQL database belongs to one of these categories.

Let's understand each one deeply.

---

# 📄 1. Document Database

## 📖 Definition

A **Document Database** stores information as **documents** instead of rows.

Each document usually uses a JSON-like structure (many systems internally use BSON or similar formats).

Unlike SQL,

there is **no requirement** that every document has the same fields.

---

# 🏠 Real-Life Analogy

Imagine a file cabinet.

Each customer has one file.

Some files contain

* Name
* Email
* Phone

Others contain

* Name
* Email
* Skills
* GitHub
* Portfolio

Every file can be different.

---

# 📄 Example

Customer A

```json
{
  "name": "Rahul",
  "email": "rahul@gmail.com"
}
```

Customer B

```json
{
  "name": "Priya",
  "email": "priya@gmail.com",
  "skills": [
    "Java",
    "SQL",
    "React"
  ]
}
```

Notice something?

Both documents are valid.

No table structure had to change.

---

# 🤔 Why Is This Useful?

Suppose tomorrow you want to add

```text
LinkedIn Profile
```

SQL

↓

Often requires altering the table schema.

Document Database

↓

Simply add the new field to new or existing documents as needed.

Much easier.

---

# 🌍 Popular Document Databases

Examples include:

* MongoDB
* CouchDB

---

# 🚀 Real-World Use Cases

Document databases are commonly used for:

📱 User Profiles

🛒 Product Catalogs

📰 CMS Systems

📦 Inventory Systems

📄 Content Management

# 🔑 2. Key-Value Database

This is one of the simplest database models.

## 📖 Definition

A **Key-Value Database** stores data as:

```text
Key

↓

Value
```

Every key is unique.

The value can be almost anything.

---

# 🏠 Real-Life Analogy

Think of a dictionary.

Word

↓

Meaning

Example

```text
Apple

↓

Fruit
```

Here

Apple

is the key.

Fruit

is the value.

---

# 💻 Example

```text
user:101

↓

Rahul
```

Another example

```text
session:ABC123

↓

Logged In
```

Database simply stores

Key

↓

Value

Nothing more.

---

# 🚀 Why Is It So Fast?

Suppose you know the exact key.

The database can retrieve the value with very few operations.

This makes key-value stores ideal for:

⚡ Session Storage

⚡ Login Tokens

⚡ Caching

⚡ Shopping Carts

---

# 🌍 Popular Key-Value Databases

Examples include:

* Redis
* Amazon DynamoDB (supports key-value and document models)

---

# 🏢 Real Company Example

Imagine Amazon.

When you log in,

the system stores your session.

Instead of repeatedly reading your user details from a relational database,

a fast key-value store may hold temporary session information.

This improves response times.

---

# 📊 3. Wide-Column Database

## 📖 Definition

A **Wide-Column Database** stores data in rows,

but unlike relational databases,

different rows can have different columns.

Columns are grouped into **column families**, allowing flexible storage.

---

# 🏠 Real-Life Analogy

Imagine a school report.

Rahul

has

Math

Science

English

---

Priya

has

Math

Science

Computer

Different students may have different subjects.

A wide-column database can naturally handle varying columns.

---

# 💻 Example

Row 1

```text
Student

↓

Name

Math

Science
```

Row 2

```text
Student

↓

Name

History

Biology

Geography
```

Both rows are valid.

---

# 🚀 Why Is It Useful?

Wide-column databases are optimized for:

📈 Massive datasets

⚡ High write throughput

🌍 Distributed storage

They are widely used in:

IoT

Analytics

Event Logging

Large-scale telemetry systems

---

# 🌍 Popular Wide-Column Databases

Examples include:

* Apache Cassandra
* Apache HBase

---

# 🏢 Real Company Example

Imagine millions of smart electricity meters sending readings every few seconds.

A wide-column database can efficiently store and distribute this time-series-like data across many servers.

---

# 🕸️ 4. Graph Database

## 📖 Definition

A **Graph Database** stores data as:

* Nodes
* Relationships (Edges)

Instead of focusing on tables,

it focuses on how things are connected.

---

# 🏠 Real-Life Analogy

Imagine Facebook.

Rahul

↓

Friend

↓

Priya

↓

Friend

↓

Ankit

Everything is connected.

A graph naturally represents these relationships.

---

# 💻 Example

```text
Rahul

↓

FRIEND

↓

Priya

↓

WORKS WITH

↓

Amit

↓

LIVES IN

↓

Delhi
```

The relationships themselves are first-class citizens.

---

# 🤔 Why Not Use SQL?

You could model friendships using tables.

But imagine

500 million users

and

100 billion friendships.

Queries like

> "Find friends of friends"

can become very expensive in relational databases.

Graph databases are optimized for traversing connected data.

---

# 🌍 Popular Graph Databases

Examples include:

* Neo4j
* Amazon Neptune

---

# 🏢 Real-World Use Cases

Graph databases are commonly used for:

👥 Social Networks

🛒 Recommendation Engines

💳 Fraud Detection

📍 Route Planning

🌐 Knowledge Graphs

# 📊 Comparison Table

| Type           | Stores Data As                     | Best For                                          | Example   |
| -------------- | ---------------------------------- | ------------------------------------------------- | --------- |
| 📄 Document    | JSON-like Documents                | User Profiles, CMS, Product Catalogs              | MongoDB   |
| 🔑 Key-Value   | Key → Value                        | Sessions, Cache, Tokens                           | Redis     |

---

# 🧠 Which One Should You Choose?

There is no universally best database.

Choose based on the problem.

### Use a Document Database when

✅ Data structure changes frequently.

---

### Use a Key-Value Database when

✅ You already know the key.

✅ Speed is the highest priority.

---

### Use a Wide-Column Database when

✅ You need to handle massive distributed datasets with high write throughput.

---

### Use a Graph Database when

✅ Relationships are the most important part of the data model.

---

# 🏢 Modern Companies Use Multiple Databases

A single company often uses several database types.

Example:

🛒 Amazon

* Customer Orders → Relational Database
* User Sessions → Key-Value Store
* Product Catalog → Document Database
* Recommendations → Graph Database

This is another example of **polyglot persistence**.

# 🎤 Interview Questions

## ❓ What are the four types of NoSQL databases?

### ✅ Answer

The four major types are:

* **Document Databases**, which store JSON-like documents.
* **Key-Value Databases**, which store data as key-value pairs.
* **Wide-Column Databases**, which organize data into flexible column families for large-scale distributed storage.
* **Graph Databases**, which model data as nodes and relationships for highly connected datasets.

---

## ❓ Which NoSQL database is best?

### ✅ Answer

There is no single best NoSQL database. The choice depends on the application's workload and access patterns. Document databases suit flexible documents, key-value databases excel at caching and sessions, wide-column databases handle massive distributed datasets, and graph databases are ideal for relationship-heavy applications.

---

# 🎯 Key Takeaways

✅ NoSQL is a family of database models, not a single database.

✅ Document databases store flexible JSON-like documents.

✅ Key-value databases provide extremely fast lookups by unique keys.

✅ Wide-column databases are optimized for large-scale distributed workloads.

✅ Graph databases efficiently model and query complex relationships.

✅ Modern applications frequently combine multiple database technologies, selecting each one based on the problem it solves best.
