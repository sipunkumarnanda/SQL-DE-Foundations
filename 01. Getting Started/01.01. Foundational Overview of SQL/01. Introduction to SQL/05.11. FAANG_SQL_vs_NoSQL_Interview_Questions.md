# 🌟 Interview Tip

One of the biggest mistakes candidates make is answering:

> **Interviewer:** "Which is better, SQL or NoSQL?"

**Candidate:**

> "MongoDB is better because it's faster."

❌ This is a weak answer.

A strong engineer first asks:

* What kind of application?
* What is the workload?
* Is consistency important?
* Are relationships important?
* What are the scalability requirements?

There is **no universally best database**.

# 🎤 Question 1

## ❓ What is the difference between SQL and NoSQL?

### ✅ Answer

SQL databases store data in related tables with a predefined schema and provide strong support for transactions, relationships, and complex queries.

NoSQL databases support multiple data models—such as documents, key-value pairs, wide columns, and graphs—and are designed for workloads requiring flexibility, horizontal scaling, or distributed architectures.

The correct choice depends on the application's requirements.

---

# 🎤 Question 2

## ❓ Which database is faster?

### ❌ Wrong Answer

> MongoDB is always faster.

---

### ✅ Correct Answer

Neither database is always faster.

Performance depends on:

* Query pattern
* Data model
* Indexes
* Hardware
* Database design
* Workload

For example:

* Complex joins and transactions are often handled efficiently by relational databases.
* Flexible document retrieval and some distributed workloads may perform better with document databases.

The right answer is:

> **"It depends on the workload."**

---

# 🎤 Question 3

## ❓ Why was NoSQL invented?

### ✅ Answer

NoSQL databases were introduced to address challenges faced by large-scale internet applications, such as flexible schemas, horizontal scaling, distributed storage, and high write throughput.

They were not designed to replace SQL databases but to solve problems where the relational model is not always the best fit.

---

# 🎤 Question 4

## ❓ Why do banks use SQL?

### ✅ Answer

Banks require:

* ACID transactions
* Strong consistency
* Reliable recovery
* Auditing
* Referential integrity

Relational databases provide these capabilities, making them well suited for financial systems.

---

# 🎤 Question 5

## ❓ Why do social media platforms often use NoSQL?

### ✅ Answer

Social media applications generate rapidly changing, media-rich, and high-volume data.

NoSQL databases offer flexible schemas and scalable architectures that suit many of these workloads.

However, many social media companies also use SQL databases for billing, user accounts, or other transactional components.

---

# 🎤 Question 6

## ❓ Can MongoDB replace MySQL?

### ✅ Answer

No.

MongoDB and MySQL solve different problems.

MySQL excels at:

* Structured data
* Transactions
* Complex relationships
* Reporting

MongoDB excels at:

* Flexible documents
* Evolving schemas
* Certain distributed workloads

Many modern systems use both.

---

# 🎤 Question 7

## ❓ Why do companies use multiple databases?

### ✅ Answer

Different parts of a system have different requirements.

For example:

```text
Orders
        ↓
      SQL

Sessions
        ↓
   Key-Value Store

Product Catalog
        ↓
   Document Database

Recommendations
        ↓
   Graph Database
```

Using multiple databases allows each service to use the technology best suited to its workload.

This approach is called **polyglot persistence**.

---

# 🎤 Question 8

## ❓ Why don't SQL databases store everything in one table?

### ✅ Answer

Storing all information in one table creates:

* Data redundancy
* Update anomalies
* Larger storage requirements
* Difficult maintenance
* Poor normalization

Relational databases split data into related tables connected through keys, improving consistency and reducing duplication.

---

# 🎤 Question 9

## ❓ Explain the three database relationship types.

### ✅ Answer

**One-to-One**

One record relates to exactly one record.

Example:

Citizen ↔ Passport

---

**One-to-Many**

One parent record relates to many child records.

Example:

Customer → Orders

---

**Many-to-Many**

Many records relate to many other records.

Example:

Students ↔ Courses

This is implemented using a junction table.

---

# 🎤 Question 10

## ❓ What is a transaction?

### ✅ Answer

A transaction is a group of database operations treated as a single unit of work.

Either:

✅ Every operation succeeds.

or

❌ All operations are rolled back.

This guarantees data consistency.

---

# 🎤 Question 11

## ❓ Why are indexes important?

### ✅ Answer

Indexes help databases locate data efficiently without scanning every row.

They work similarly to the index in a book, allowing the database to find the required information much faster.

---

# 🎤 Question 12

## ❓ What happens internally when a SELECT query is executed?

### ✅ Answer

A simplified execution flow is:

```text
SQL Query

↓

Parser

↓

Semantic Validation

↓

Query Optimizer

↓

Execution Plan

↓

Index Lookup

↓

Storage Engine

↓

Buffer Cache / Disk

↓

Result
```

Each stage helps ensure the query is valid and executed efficiently.

---

# 🎤 Question 13

## ❓ Explain ObjectId in MongoDB.

### ✅ Answer

ObjectId is MongoDB's default unique identifier.

It is a 12-byte value containing components such as:

* Timestamp
* Random value
* Counter

This allows unique IDs to be generated efficiently across distributed systems.

---

# 🎤 Question 14

## ❓ Explain Horizontal Scaling.

### ✅ Answer

Horizontal scaling means increasing capacity by adding more servers instead of making a single server more powerful.

```text
One Server

↓

Two Servers

↓

Four Servers

↓

Eight Servers
```

Many NoSQL databases are designed with horizontal scaling in mind.

---

# 🎤 Question 15

## ❓ Explain Vertical Scaling.

### ✅ Answer

Vertical scaling means upgrading a single server by adding more CPU, RAM, or storage.

```text
Small Server

↓

Bigger Server

↓

More Powerful Server
```

It is simpler initially but has practical hardware limits.

# 🎤 Question 16 (Scenario)

## ❓ You're building a Banking Application. Which database would you choose?

### ✅ Answer

I would choose a relational database because banking requires:

* ACID transactions
* Strong consistency
* Referential integrity
* Rollback support
* Auditing
* Reliable recovery

These are core strengths of SQL databases.

# 🎤 Question 17 (Scenario)

## ❓ You're building Instagram. Which database would you choose?

### ✅ Answer

I would likely use multiple databases.

For example:

* User accounts → SQL
* User profiles and content → Document Database
* Session storage → Key-Value Store
* Recommendations → Graph Database

Different components have different requirements.

---

# 🎤 Question 18 (Scenario)

## ❓ You're building an E-Commerce Platform.

### ✅ Answer

I would use a combination of databases.

Orders and payments require SQL because they need transactions and consistency.

Product catalogs may use a document database because products often have different attributes.

Caching can use a key-value store.

Recommendations may use a graph database.

---

# 🎤 Question 19

## ❓ What is Polyglot Persistence?

### ✅ Answer

Polyglot persistence is the practice of using multiple database technologies within the same application.

Each database is selected based on the workload it handles best.

This is common in large-scale distributed systems.

---

# 🎤 Question 20

## ❓ If SQL is so good, why doesn't every company use only SQL?

### ✅ Answer

Because different workloads have different requirements.

Relational databases are excellent for structured, transactional data.

Some applications benefit more from flexible schemas, distributed storage, or specialized data models.

Modern companies often combine SQL and NoSQL technologies rather than choosing only one.

# 🧠 Top 10 Mistakes Beginners Make

❌ Thinking MongoDB is always faster.

❌ Thinking SQL is outdated.

❌ Using one database for every problem.

❌ Ignoring indexes.

❌ Ignoring transactions.

❌ Confusing horizontal and vertical scaling.

---

# 🎯 Key Takeaways

✅ Database selection is a design decision, not a popularity contest.

✅ There is no universally "best" database.

✅ SQL excels at structured, transactional, relationship-heavy workloads.

✅ NoSQL excels at flexible schemas, distributed architectures, and certain large-scale workloads.

✅ Large companies commonly use multiple database technologies through polyglot persistence.

✅ In interviews, always justify your choice using the application's requirements and trade-offs rather than making absolute claims.
