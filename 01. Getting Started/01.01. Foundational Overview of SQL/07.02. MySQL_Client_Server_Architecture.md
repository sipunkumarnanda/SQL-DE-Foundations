# 🧑‍💻 Client Side

The client is any application that sends SQL queries to MySQL.

Examples:

* MySQL Workbench
* MySQL Shell
* Command Line Client
* Node.js Application
* Java Spring Boot Application
* Python Flask/Django Application
* PHP Application

The client **does not execute SQL**.

It only sends SQL statements to the server.

---

# 🏢 MySQL Server

The MySQL Server is the "brain" of the system.

It is responsible for:

* Accepting client connections
* Authenticating users
* Parsing SQL
* Optimizing queries
* Executing statements
* Managing transactions
* Reading and writing data
* Returning results

Think of it as the manager coordinating all internal components.

---

# 🔌 Connection Manager

## 📖 What is it?

The Connection Manager is the first component that receives requests from clients.

Its responsibilities include:

* Accepting new connections
* Managing active client sessions
* Tracking connection state
* Allocating server resources

# 🔐 Authentication & Authorization

After a connection request arrives, MySQL verifies:

### Authentication

"Who are you?"

It checks:

* Username
* Password

If valid, the user is authenticated.

---

### Authorization

"What are you allowed to do?"

Examples:

User A

✅ SELECT

✅ INSERT

❌ DROP TABLE

Different users can have different permissions.

# 📝 SQL Parser

Suppose you write:

```sql
SELECT name FROM employees;
```

The SQL Parser checks:

* Is the syntax correct?
* Are keywords spelled correctly?
* Are parentheses balanced?
* Is the query grammatically valid?

If you write:

```sql
SELEC name FROM employees;
```

The parser immediately returns a syntax error because `SELEC` is not a valid SQL keyword.

The parser does **not** execute the query—it only validates and converts it into an internal representation.

---

# 🧠 Query Optimizer

This is one of MySQL's smartest components.

Suppose there are two ways to retrieve data:

1. Scan the entire table.
2. Use an index.

The optimizer estimates the cost of each option and chooses the one expected to be most efficient.

It may decide:

* Use an index.
* Change the join order.
* Push filters earlier.
* Eliminate unnecessary work.

Its goal is to reduce execution time and resource usage.

---

# ⚙️ Execution Engine

Once the optimizer produces an execution plan, the Execution Engine carries it out.

It:

* Requests rows from the Storage Engine.
* Applies filters.
* Performs joins.
* Groups data.
* Sorts results if required.
* Sends the final output back to the client.

Think of the optimizer as the planner and the execution engine as the worker following the plan.

---

# 💾 Storage Engine

The Storage Engine is responsible for physically storing and retrieving data.

MySQL supports multiple storage engines, including:

* InnoDB (default)
* MyISAM
* MEMORY
* CSV

The most commonly used is **InnoDB** because it supports:

* ACID transactions
* Row-level locking
* Crash recovery
* Foreign keys

---

# 🏦 What is InnoDB?

InnoDB is the default storage engine in modern MySQL.

It handles:

* Reading rows
* Writing rows
* Managing indexes
* Locking
* Transactions
* Recovery after crashes

Most production MySQL applications use InnoDB.

---

# 🧠 Buffer Pool

Reading directly from disk is much slower than reading from memory.

To improve performance, InnoDB maintains a **Buffer Pool**.

It stores frequently accessed:

* Data pages
* Index pages

Workflow:

```text
SQL Query

↓

Buffer Pool

↓

Found?

├── Yes → Return from Memory ✅
└── No → Read from Disk → Store in Buffer Pool → Return
```

Because RAM is much faster than disk storage, this significantly speeds up repeated queries.

---

# 📝 Redo Log

Imagine a transaction updates a customer's account balance.

Before the modified data is permanently written to disk, InnoDB records the change in the **Redo Log**.

Purpose:

* Recover committed transactions after a crash.
* Ensure durability (the "D" in ACID).

If the server crashes before data pages are flushed, the Redo Log helps replay committed changes during recovery.

---

# ↩️ Undo Log

Suppose a transaction starts but later fails.

Example:

```text
BEGIN;

UPDATE accounts
SET balance = balance - 1000
WHERE id = 1;

-- Error occurs

ROLLBACK;
```

The Undo Log stores enough information to reverse uncommitted changes.

It is also used by InnoDB's MVCC (Multi-Version Concurrency Control) to provide consistent reads.

---

# 📒 Binary Log (Binlog)

The Binary Log records database changes in a sequence.

Common uses:

* Replication
* Point-in-time recovery
* Backup strategies

Unlike the Redo Log, which is primarily for crash recovery within the server, the Binary Log is mainly used for replication and recovery operations.

# 🔄 Complete Query Execution Flow

Let's put everything together.

Suppose the client sends:

```sql
SELECT * FROM employees WHERE id = 101;
```

Internal flow:

```text
Client
   │
   ▼
Connection Manager
   │
   ▼
Authentication
   │
   ▼
SQL Parser
   │
   ▼
Query Optimizer
   │
   ▼
Execution Engine
   │
   ▼
Storage Engine (InnoDB)
   │
   ▼
Buffer Pool
   │
   │
   ├── Data Found in Memory?
   │         │
   │         ├── Yes → Return Rows
   │         │
   │         └── No
   │
   ▼
Read Data from Disk
   │
   ▼
Store in Buffer Pool
   │
   ▼
Return Result to Client
```

For write operations (`INSERT`, `UPDATE`, `DELETE`), additional components like the Redo Log, Undo Log, and Binary Log participate in ensuring durability, rollback capability, and replication.

# 🌍 Real-World Example

Imagine an online shopping application.

1. A customer places an order.
2. The application sends an `INSERT` query.
3. MySQL authenticates the user.
4. The parser validates the SQL.
5. The optimizer creates an execution plan.
6. The execution engine performs the insert.
7. InnoDB updates the Buffer Pool.
8. The Redo Log records the committed change.
9. The Binary Log records the event for replication.
10. The application receives a success response.

All of this typically happens in milliseconds.

# 🎤 FAANG Interview Questions

## ❓ What is the role of the SQL Parser?

### ✅ Answer

The SQL Parser checks whether the SQL statement is syntactically valid and converts it into an internal representation for further processing.

---

## ❓ What does the Query Optimizer do?

### ✅ Answer

The Query Optimizer evaluates different execution strategies, estimates their costs, and selects an efficient execution plan to minimize query execution time.

---

## ❓ Why is the Buffer Pool important?

### ✅ Answer

The Buffer Pool caches frequently accessed data and index pages in memory, reducing disk I/O and improving query performance.

---

## ❓ What is the difference between the Redo Log and the Undo Log?

### ✅ Answer

The Redo Log records committed changes so they can be replayed during crash recovery, ensuring durability. The Undo Log stores previous versions of data so uncommitted changes can be rolled back and consistent reads can be provided.

---

## ❓ What is the purpose of the Binary Log?

### ✅ Answer

The Binary Log records database changes for replication and point-in-time recovery. It is different from the Redo Log, which is primarily used for crash recovery within the server.
