Analytical databases organize data differently from OLTP databases.

Instead of focusing on transactions,

This is where **Star Schema** and **Snowflake Schema** come in.

---

# 🤔 Why Not Use the OLTP Database Structure?

Imagine an OLTP database.

You may have tables like:

```text
Customers

Orders

Products

Payments

Categories

Cities

Countries

Employees

Suppliers
```

These tables are highly normalized.

To generate a report,

the database might join many tables.

Example:

```text
Orders

↓

Customers

↓

Cities

↓

Countries

↓

Products

↓

Categories
```

Many joins increase query complexity and execution time.

For analytics,

we want fewer joins and faster reporting.

---

# 📖 Fact Table

## Definition

A **Fact Table** stores the **measurable business events**.

It contains numbers that businesses want to analyze.

Typical measures include:

* Sales Amount
* Quantity Sold
* Profit
* Discount
* Revenue

Think of it as:

> **The table that answers "What happened?"**

---

# 🏪 Example

Sales Fact Table

| Sale ID | Product ID | Customer ID | Date ID | Store ID | Quantity | Sales Amount |
| ------- | ---------- | ----------- | ------- | -------- | -------- | ------------ |
| 101     | P01        | C01         | D01     | S01      | 2        | ₹50,000      |
| 102     | P03        | C02         | D01     | S02      | 1        | ₹20,000      |

Notice something.

The Fact Table stores:

* IDs
* Numbers

Very little descriptive information.

---

# 📖 Dimension Table

## Definition

A **Dimension Table** stores descriptive information about the facts.

Think of it as:

> **The table that answers "Who?", "What?", "Where?", and "When?"**

---

| Customer ID | Name  | City   | State       |
| ----------- | ----- | ------ | ----------- |
| C01         | Rahul | Mumbai | Maharashtra |
| C02         | Priya | Delhi  | Delhi       |

---

Product Dimension

| Product ID | Product | Category    |
| ---------- | ------- | ----------- |

| P02        | Phone   | Electronics |

---

Date Dimension

| Date ID | Day | Month | Year |
| ------- | --- | ----- | ---- |
| D01     | 10  | July  | 2026 |

| Store ID | Store Name   | City   |
| -------- | ------------ | ------ |

# ⭐ What is a Star Schema?

## Definition

A **Star Schema** is a Data Warehouse design where **one central Fact Table is directly connected to multiple Dimension Tables**.

It looks like a star.

---

# ⭐ Structure

```text
                     Date

                       │

                       │

Customer ─── Sales Fact ─── Product

                       │

                       │

                    Store
```

The Fact Table sits at the center.

Dimension Tables surround it.

---

# 🌍 Why is it Called a Star?

Because visually,

it resembles a star.

```text
               Product

                  │

Customer ── Fact ── Store

                  │

                 Date
```

---

# 🚀 Advantages of Star Schema

✅ Very simple to understand.

✅ Fewer joins.

✅ Faster reporting.

✅ Excellent query performance.

✅ Easy for BI tools.

---

# ❌ Disadvantages

Because descriptive information is stored directly in dimensions,

some data may be duplicated.

Example:

```text
City

↓

State

↓

Country
```

The same country name may appear many times.

This increases storage.

---

# ❄️ What is a Snowflake Schema?

Instead of storing everything in one Dimension Table,

the dimensions are normalized into additional tables.

---

# Structure

```text
                    Country

                       │

                    State

                       │

                     City

                       │

Customer ─── Sales Fact ─── Product

                       │

                     Date
```

Notice something.

The Customer Dimension has been divided into smaller related tables.

---

# Why is it Called Snowflake?

The dimensions branch out.

Visually,

it resembles a snowflake.

---

# 🚀 Advantages

✅ Less duplicated data.

✅ Better normalization.

✅ Lower storage usage.

✅ Easier to maintain shared reference data.

---

# ❌ Disadvantages

More joins are required.

Queries become more complex.

Analytical performance may be slower than a Star Schema for some workloads.

# 📊 Star vs Snowflake

| Feature         | Star Schema             | Snowflake Schema      |
| --------------- | ----------------------- | --------------------- |
| Structure       | Denormalized Dimensions | Normalized Dimensions |
| Number of Joins | Fewer                   | More                  |
| Query Speed     | Usually Faster          | Usually Slower        |
| Storage         | More                    | Less                  |
| Complexity      | Simple                  | Higher                |
| Maintenance     | Easier                  | More Structured       |

---

# 🤔 Which One is Better?

There is no universal answer.

Use **Star Schema** when:

* Reporting speed is the priority.
* Simplicity matters.
* BI dashboards are heavily used.

Use **Snowflake Schema** when:

* Reducing redundancy is important.
* Dimension data is complex.
* Storage efficiency matters more.

Many real-world data warehouses use a combination of both approaches.

# 🎤 FAANG Interview Questions

## ❓ What is a Fact Table?

### ✅ Answer

A Fact Table stores measurable business events such as sales, revenue, quantity, or profit. It usually contains numeric measures along with foreign keys that reference related Dimension Tables.

---

## ❓ What is a Dimension Table?

### ✅ Answer

A Dimension Table stores descriptive information about business entities, such as customers, products, dates, or locations. It provides context for analyzing the facts.

---

## ❓ What is a Star Schema?

### ✅ Answer

A Star Schema is a Data Warehouse design where one central Fact Table connects directly to multiple denormalized Dimension Tables. It minimizes joins and is optimized for fast analytical queries.

---

## ❓ What is a Snowflake Schema?

### ✅ Answer

A Snowflake Schema is a variation of the Star Schema in which Dimension Tables are normalized into additional related tables. This reduces data redundancy but increases the number of joins.

---

## ❓ Which is faster, Star or Snowflake?

### ✅ Answer

Star Schema is generally faster for analytical queries because it requires fewer joins. Snowflake Schema reduces redundancy and improves normalization but may require additional joins during query execution.

---

# 🎯 Key Takeaways

✅ Fact Tables store measurable business events.

✅ Dimension Tables store descriptive business information.

✅ Star Schema places one Fact Table at the center with directly connected Dimension Tables.

✅ Snowflake Schema normalizes Dimension Tables into additional related tables.

✅ Star Schema typically favors query performance and simplicity.

✅ Snowflake Schema favors reduced redundancy and structured dimension data.
