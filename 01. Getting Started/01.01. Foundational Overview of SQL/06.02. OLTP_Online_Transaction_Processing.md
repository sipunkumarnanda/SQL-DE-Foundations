# 📖 What is OLTP?

## Definition

**OLTP (Online Transaction Processing)** is a type of database workload designed to process **large numbers of short, fast, and reliable transactions in real time**.

Simply put:

> **OLTP systems are responsible for running the daily operations of a business.**

# ⚡ Characteristics of OLTP

OLTP systems have several important characteristics.

---

## 1️⃣ Large Number of Users

Imagine Amazon during a festival sale.

Millions of users are:

* Browsing products
* Buying products
* Making payments

The database must serve all of them simultaneously.

---

## 2️⃣ Small Queries

Typical OLTP queries affect only a small amount of data.

Example

```sql
SELECT *
FROM Accounts
WHERE account_id = 101;
```

Only one account is being read.

---

Another example

```sql
UPDATE Accounts
SET balance = balance - 500
WHERE account_id = 101;
```

Again,

only one row changes.

---

# 3️⃣ Very Fast Response Time

Imagine withdrawing money from an ATM.

Would you wait

5 minutes?

No.

Expected response:

⚡ Less than a second.

OLTP systems are optimized for low latency.

---

# 4️⃣ High Concurrency

Concurrency means

**many users accessing the database at the same time**.

Example

Amazon

```text
Customer A

↓

Places Order

Customer B

↓

Cancels Order

Customer C

↓

Makes Payment

Customer D

↓

Updates Address
```

All these operations happen simultaneously.

The database must ensure they do not interfere with one another.

---

# 5️⃣ ACID Transactions

Every transaction must be:

✅ Atomic

✅ Consistent

✅ Isolated

✅ Durable

Without ACID,

financial systems would become unreliable.

(We covered ACID in Chapter 5.)

# 🏗️ Internal Architecture of an OLTP System

When a user performs an operation,

the request passes through several stages.

```text
User

↓

Application

↓

SQL Query

↓

Transaction Manager

↓

Lock Manager / MVCC

↓

Storage Engine

↓

Transaction Log (WAL)

↓

Database Files

↓

Commit

↓

Response to User
```

Each component has a specific responsibility.

Let's understand them.

---

# 🧠 Step 1 — User Request

Suppose Rahul transfers

₹5,000

to Priya.

The application sends the request to the database.

---

# 📝 Step 2 — Transaction Begins

The database starts a transaction.

```sql
BEGIN;
```

At this point,

no permanent change has been made.

---

# 🔒 Step 3 — Locking or MVCC

The database ensures that two users cannot modify the same data in conflicting ways.

Depending on the database,

this is achieved using:

* Row Locks
* Table Locks
* MVCC (Multi-Version Concurrency Control)

This prevents inconsistent results.

---

# 📄 Step 4 — Transaction Log (Write-Ahead Logging)

Before modifying the actual data,

the database writes the intended changes to a **transaction log**.

Example:

```text
Transfer ₹5000

Account A → -5000

Account B → +5000
```

Only after the log is safely written does the database update the data.

This allows recovery after crashes.

---

# 💾 Step 5 — Update Data

Now the database updates:

* Account A
* Account B

The changes are applied to the database pages.

---

# ✅ Step 6 — Commit

If everything succeeds,

the database executes:

```sql
COMMIT;
```

Now the changes become permanent.

---

# ❌ Step 7 — Rollback (If Needed)

Suppose the server crashes after deducting money from Account A but before crediting Account B.

The database executes:

```sql
ROLLBACK;
```

The transaction is undone.

The customer never loses money.

---

# 🏦 Banking Example

Rahul transfers ₹10,000 to Priya.

Steps:

```text
BEGIN

↓

Deduct ₹10,000

↓

Credit ₹10,000

↓

Write Transaction Log

↓

COMMIT

↓

Success
```

If any step fails:

```text
ROLLBACK
```

Everything returns to the original state.

# 🎤 FAANG Interview Questions

## ❓ What is OLTP?

### ✅ Answer

OLTP (Online Transaction Processing) is a database workload designed for handling large numbers of short, fast, and reliable transactions in real time. It supports day-to-day business operations such as banking transactions, online shopping, ticket booking, and attendance systems.

---

## ❓ Why are OLTP queries small?

### ✅ Answer

OLTP systems are optimized for real-time operations. Most transactions involve only a small number of rows, allowing the database to respond quickly while supporting many concurrent users.

---

## ❓ Why is ACID important in OLTP?

### ✅ Answer

ACID ensures that transactions are reliable. It guarantees that operations either complete entirely or not at all, maintains data consistency, isolates concurrent transactions, and ensures committed changes survive failures.

---

## ❓ Why do OLTP systems need indexes?

### ✅ Answer

Indexes allow the database to locate rows efficiently instead of scanning entire tables. This reduces query response time, which is critical for real-time transactional systems.

---

# 🎯 Key Takeaways

✅ OLTP handles the daily operational workload of a business.

✅ It processes many short, fast transactions.

✅ It supports high concurrency with many simultaneous users.

✅ ACID transactions are fundamental to OLTP reliability.

✅ Transaction logs, locking/MVCC, indexes, and buffer caches all contribute to OLTP performance.

✅ Examples include banking systems, e-commerce platforms, ride-sharing apps, messaging apps, and college management systems.
