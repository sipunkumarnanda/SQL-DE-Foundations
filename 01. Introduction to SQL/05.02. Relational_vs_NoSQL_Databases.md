# 🚨 Problems Caused by Data Redundancy

Repeating data creates many problems.

Imagine Rahul changes his email.

Old email

```text
rahul@gmail.com
```

New email

```text
rahul123@gmail.com
```

Now suppose Rahul has

500 attendance records.

You must update

500 rows.

If you forget even one row,

your database now contains two different emails for the same student.

Which one is correct?

Nobody knows.

---

# ❌ This Leads to Data Inconsistency

Data inconsistency means

> The same real-world information exists in multiple places but contains different values.

Example

Attendance Table

| Student | Email                                     |
| ------- | ----------------------------------------- |
| Rahul   | [rahul@gmail.com](mailto:rahul@gmail.com) |

Fees Table

| Student | Email                                           |
| ------- | ----------------------------------------------- |
| Rahul   | [rahul123@gmail.com](mailto:rahul123@gmail.com) |

Now your system has conflicting information.

Large organizations cannot afford such errors.

---

# 💡 The Solution: Divide Data

Instead of one huge table,

we divide information into multiple smaller tables.

Example

```text
Database

│

├── Students

├── Teachers

├── Courses

├── Attendance

├── Fees

├── Subjects
```

Each table stores only one type of information.

# 🤔 But How Does the Database Know Which Attendance Belongs to Rahul?

This is where the word

# 🔗 Relation

comes into the picture.

The Attendance table stores

```text
Student ID = 101
```

The Students table also stores

```text
Student ID = 101
```

That common value creates a connection.

```text
Students

101 Rahul

↓

Student ID

↓

Attendance

101 Present
101 Present
101 Absent
```

This connection is called a **relationship**.

---

# 📖 What is a Relation?

Mathematically,

a **Relation** is a set of tuples (rows) with the same attributes (columns).

That definition comes from relational theory.

For practical software development,

it's easier to think of it this way:

> A **relation** is a logical connection between related pieces of data.

When two tables share a common key,

the database understands how they are connected.

---

# 🌳 Why is it Called a Relational Database?

Because it stores data in

✅ Tables

and

✅ Connects those tables using relationships.

The relationships allow the database to reconstruct complete information whenever needed.

Example

```text
Students

↓

Student ID

↓

Attendance

↓

Fees

↓

Courses

↓

Exams
```

Everything is connected.

That is why it is called a **Relational Database**.

---

# 🏛️ The Relational Model

In 1970,

computer scientist **Edgar F. Codd** proposed the **Relational Model**.

His idea was revolutionary.

Instead of storing data as disconnected files,

store it as **related tables** based on mathematical principles.

The model introduced concepts such as:

* Relations (Tables)
* Tuples (Rows)
* Attributes (Columns)
* Keys
* Constraints

Nearly every modern SQL database is built on these ideas.

# 🚀 Advantages of Relationships

Because tables are connected,

the database gains many benefits.

### ✅ Less Duplicate Data

Store customer information once.

Reuse it everywhere.

---

### ✅ Better Consistency

If Rahul changes his email,

update it in one place.

Every related table automatically points to the updated record.

---

### ✅ Easier Maintenance

Smaller tables are easier to understand, maintain, and optimize.

---

### ✅ Better Performance

Smaller, focused tables often reduce unnecessary duplication and improve storage efficiency.

---

### ✅ Strong Data Integrity

Relationships help prevent invalid references and maintain consistency across tables.

# ❓ Common Beginner Misconception

### "Does a Relational Database Mean Every Table Must Be Connected?"

❌ No.

A database can contain independent tables.

However,

whenever data naturally relates to other data,

relational databases allow you to model those relationships cleanly and efficiently.

# 🎤 FAANG Interview Questions

## ❓ What is a Relational Database?

### ✅ Answer

A relational database is a database management system that stores data in tables and organizes related information through relationships between those tables. These relationships are typically established using keys, allowing data to remain consistent, organized, and easy to query.

---

## ❓ Why is it called a Relational Database?

### ✅ Answer

It is called a relational database because it is based on the relational model, where data is represented as relations (tables) and related tables are connected through common keys. These relationships allow the database to combine information from multiple tables while minimizing redundancy and maintaining data integrity.

---

## ❓ Why don't we store everything in one table?

### ✅ Answer

Storing everything in one table leads to data redundancy, inconsistency, larger storage requirements, and difficult maintenance. Splitting data into related tables reduces duplication, improves consistency, simplifies updates, and makes the database easier to scale and manage.

---

# 🎯 Key Takeaways

✅ A relational database stores data in tables.

✅ Each table represents one type of real-world entity.

✅ Tables are connected using relationships based on common keys.

✅ Relationships reduce redundancy and improve consistency.

✅ The relational model was proposed by **Edgar F. Codd** and forms the foundation of modern SQL databases.

✅ Relational databases are ideal for applications that require structured data, strong consistency, and reliable relationships between entities.
