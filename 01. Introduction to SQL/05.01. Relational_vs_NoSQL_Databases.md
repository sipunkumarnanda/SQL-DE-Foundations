# 🧠 Understanding Different Types of Data

Before understanding databases,

we need to understand the kinds of data they store.

There are three major categories.

---

# 📋 1. Structured Data

Structured data follows a fixed format.

Every record has the same columns.

Example

Students Table

| Student ID | Name  | Email                                     | Fees  |
| ---------- | ----- | ----------------------------------------- | ----- |
| 101        | Rahul | [rahul@gmail.com](mailto:rahul@gmail.com) | 25000 |
| 102        | Priya | [priya@gmail.com](mailto:priya@gmail.com) | 22000 |

Notice something?

Every row has exactly the same structure.

This type of data is predictable.

SQL databases are excellent at managing structured data.

---

# 📄 2. Semi-Structured Data

Semi-structured data has some organization,

but not a fixed table structure.

Example JSON:

```json
{
  "name": "Rahul",
  "email": "rahul@gmail.com",
  "skills": [
    "Java",
    "SQL",
    "React"
  ]
}
```

Now another user

```json
{
  "name": "Priya",
  "phone": "9876543210",
  "experience": 3
}
```

Notice that the fields are different.

There is no fixed set of columns.

This flexibility is useful when data varies between records.

Document-oriented NoSQL databases, such as MongoDB, are designed for this style of data.

---

# 🎥 3. Unstructured Data

Some information doesn't naturally fit into rows and columns.

Examples:

📸 Images

🎥 Videos

🎵 Audio

📄 PDFs

📧 Emails

📱 Social Media Posts

These are examples of unstructured data.

Typically, databases store metadata about these files (such as file names or URLs), while the actual files are stored in dedicated file or object storage systems.

# 🚀 The Evolution of Databases

Let's see how databases evolved over time.

### 1960s

Data stored in files.

Problems:

❌ Duplicate data

❌ Difficult searching

❌ No relationships

---

### 1970

The **Relational Model** was proposed by computer scientist Edgar F. Codd.

The idea was revolutionary.

Instead of random files,

store data in related tables.

This became the foundation of relational databases.

---

### 1980–2005

Relational databases became the industry standard.

Popular systems included:

* Oracle Database
* MySQL
* PostgreSQL
* Microsoft SQL Server

Almost every business relied on SQL databases.

---

### Around 2005 Onwards

The internet changed everything.

Companies like

* Google
* Amazon
* Meta
* Netflix

began serving hundreds of millions of users.

Traditional relational databases were still excellent, but some new workloads required greater flexibility and easier horizontal scaling.

This led to the rise of NoSQL databases for specific use cases.

---

# 🌍 Why Was NoSQL Invented?

Many people think

> "SQL became old."

❌ Wrong.

SQL is still one of the most important technologies in software engineering.

NoSQL was **not** invented to replace SQL.

It was invented to solve different problems.

For example,

Imagine Instagram.

One user may have:

* Name
* Email
* Bio

Another user may also have:

* Website
* Business Category
* Verification Badge
* Shopping Links

The data structure can evolve frequently.

Using a rigid table schema for every possible variation can become inconvenient.

Document databases allow more flexible document structures.

---

Another challenge is scale.

Imagine:

```text
Google Search

↓

10 Billion+ searches

↓

Every day
```

Systems operating at this scale often distribute data across many servers.

Many NoSQL databases are designed with horizontal scaling as a primary goal.

---

# ⚖️ SQL vs NoSQL Philosophy

Relational Databases focus on

✅ Structure

✅ Consistency

✅ Relationships

✅ Data Integrity

---

NoSQL Databases focus on

⚡ Flexibility

⚡ Scalability

⚡ Fast access for certain workloads

⚡ Handling evolving data models

Neither philosophy is universally better.

Each is optimized for different kinds of problems.

# 🏢 Real Company Examples

### 🏦 Banking

Priority:

🔒 Accuracy

Choice:

Relational Database

---

### 💬 Chat Application

Priority:

⚡ High throughput

⚡ Rapid message handling

Many chat systems use a mix of technologies, often including NoSQL databases for message storage and SQL databases for account and billing data.

---

### 🛒 Amazon

Amazon doesn't rely on just one database.

Different services use different databases depending on the workload.

This approach is known as **polyglot persistence**—using multiple database technologies where each one fits best.

---

# 🎤 FAANG Interview Question

## ❓ Why were NoSQL databases invented if SQL databases already existed?

### ✅ Best Answer

> NoSQL databases were created to address challenges that became prominent with large-scale internet applications, such as flexible schemas, massive horizontal scaling, and high-throughput workloads. Relational databases remain the best choice for applications requiring strong consistency, complex relationships, and transactional integrity. Rather than replacing SQL databases, NoSQL databases complement them by solving different categories of problems.

---

# 🎯 Key Takeaways

✅ Every application generates data, but different applications generate different kinds of data.

✅ The primary challenge is not storing data—it is storing, organizing, retrieving, and scaling it efficiently.

✅ Structured, semi-structured, and unstructured data each have different characteristics and storage needs.

✅ Relational databases excel at structured data, relationships, consistency, and transactions.

✅ NoSQL databases provide flexibility and are often well-suited for rapidly changing schemas and large-scale distributed workloads.

✅ Modern technology companies frequently use **multiple database technologies together**, choosing the right database for each specific problem rather than relying on a single solution.
